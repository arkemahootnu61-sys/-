# labuladong算法网站学习笔记

## 【1】LLM以及agent的核心术语以及原理

- LLM与agent之间的关系就好比与汽车的引擎以及汽车之间的管理，agent就相当于LLM的工程实现
- LLM本质上是一个概率预测，我们通常AI回答的时候看到字一个一个往外蹦，其本质上就是他在一个一个预测概率，因为参数高达几百上千亿，所以这个预测的概率就高的惊人
  - 这里就是有一个温度的概念，当温度为0是就会按照概率最大的进行生成，比较适用于编程等严谨的工作，但是如果是比较有创意的工作，温度就会调的比较高
- token就是输入以及输出的文本，是模型处理的最小单位，所有的聊天记录就构成了上下文，但是上下文的窗口是有限的，怎么解决这个问题，一个就是压缩，另外一个就是渐进式输出

### base模型以及instruct模型

- base模型单纯就是一个文本补全的概率模型，你问他一个问题，他只会一直补全，但是instruct通过大量的指令以及回答来对数据进行微调
- RLHF是基于人类的反馈强化学习，就是让AI自己回答问题，然后人类对他的回答进行打分，然后让AI自己进行反思总结，调整自己想法

### 思维链

- 思维链：我们人类在写题的时候写出来过程会提高正确率，AI也一样，像deepseek就是第一家把思维链写出来的工具，有了思维链，后学的推导就会有更高的正确率
  - 我们有时候写题的时候会进行反思，AI有时候也会进行自我反思，本质上就是用更多的时间来换取更高精确度，这就是不同模式之间的区别，思考的快慢，之前都是单一模式以及强度参数的思考，后来才加入这种模式，可以更加省心。

### 系统提示词

- system prompt来设置人设，经常有时候我们聊一些不好的东西，他不会跟你聊天，这是因为工程师会给他提前做了一些设定，设置安全规则，还会有tools的API
  - 用户的优先级是小于工程师的，所以我们有时候跟他说，上面的都不管用的时候，根本没有用，但是网上有这种专门越狱的.

### tools

- LLM不会上网，其本质上还有一个大脑，有很多工具他是不会调用的，所以工程师们给他写了很多工具的API接口来到system prompt，当LLM发现自己有这个工具，就会返回tool的API，然后你的系统会去调用工具，最后返回的喂给LLM，最后在回答你的问题，这就是为什么，你只是说一个hi就会消耗很多的tokens，但是后面就没有那么多了，根本原因就是第一次还发送了人设，安全规则，以及很多的tools的API，所以会耗费很多的tokens，**模型输出结构化的工具，外围程序解析结构化的工具，执行后喂给上下文**
  - 对于不同的AI，以及不同的工具，可能每一个工具的接口都不一样，就像当年的充电线一样，那怎么办呢，于是anthropic推出了自己的MCP（model context protocol,模型上下文协议）来统一规范接口的形式
  - MCP是为所有的tools接口提供了统一的规范标准，我们可以MCP来开发客户端与服务端，但是与传统的客户端与服务端不同的是客户端是你的AI编辑器，服务端是你的装满工具的AI代码服务器
    - 服务端上面有很多的tools，无论是code X还是claude都可以调用，第一次先发送请求，然后服务端返回自己工具的集合，如果发生变更，服务端会发送推送
    - MCP支持两种协议，分别是stdio以及http
      - stdio是服务器以及客户端是在同一个机器，标准输入输出来进行传输
      - http是远程传输，通过远程进行传输
      - stdio就是启动服务，http就是连接服务
      - 目前服务端也可以向服务端索信息，这个就是elicitation，
        - 比如说，把codex做成claude的这个客户端，把它审查代码的功能包装成tools的接口，那么有一任务需要改动你的代码，那你准备怎么做呢，这个时候就需要人工来决定你需不需要来进行执行，这个就需要客户端向服务端发送请求
        - elicitation一般是询问是否同意或者拒绝，或者取消，但是还有一种表单模式，会询问你一节具体的东西，但是一些敏感的东西他不会询问你，怎么判定呢，就看黑客拿到了这行东西会不会损坏你的钱包，会不会拿走你的数据那就必须走URL模式
        - URL模式就是它给你提供一个网页，然后让你在网页上自己去填充，在这个过程中，客户端以及LLM都只能够看到URL,不能够看到你输入的敏感信息
        - TLS是一种加密技术，TCP三次握手手以及四次挥手，三次挥手就相当于是建立连接，TLS相当于对暗号，确定连接正确
  - 用MCP查天气的工具，用fastmcp就可以搞定
- **所以，agent本质上看来就把LLM + tool use加上循环**，模型在循环中不断思考,行动，观察结果，继续思考，这个模式学术山叫做ReAct（Reasoning + Acting），就是先不知道怎么做，先看看情况，再看下一步怎么做，

### skill

- 对于一些重复使用的流程，比如说我让agent识别图片整理成文档，就可以整理成文档，并且说明使用情景，下载遇到类似的情况就可以直接无脑调用了
  - 渐进式披露，通俗来讲，就是现在AI加载一个目录，说明了我有哪些skill而不全部加载，等到要使用的时候在加载进来，这样既能节省tokens

### RAG检索生成增强

- 加入你给AI一个学生手册，然后向他询问一些问题，但是你问的问题不一定跟手册的一摸一样，而且手册的关键词也有很多，那么AI是精确的理解你需要找的是什么呢，这就需要用到embedding模型了，他会将你的每一句话转化成一组语义向量（一组浮点数，比如千问就有1024维度），语义相近的文本向量距离也很近
- 通过对比学习，通过海量的句子训练，模型会有一套模型编码器，没有见过新的句子也能给出语意向量
- 数据分块的技巧，第一就是永远不破坏句子的边界，其次就是前一句切的跟后一句切的会有一些重复，这样就可以让他也能够理解上下句子的语意连接
- 编程agent是不用RAG的，使用的是grep进行精准搜索
  - 索引容易过期
  - 代码语言和自然语言不一样
  - 安全和意思顾虑
  - 所以现在有一种模式就是先用RA G搜索一下，接着再使用grep进行搜索


### 自动记忆：跨会话的信息延续

- 有时候我们新开会话，但是有一些内容是需要记忆的，分为两个维度
  - 用户记忆，比如你的生日，你的风格喜好
  - 项目记忆，一些项目的坑，规范的风格，以及接口等等等
  - **总之，就是把跨对话有价值的文本沉淀下来，新的对话自动加载到上下文**

### harness(马具)：让agent更加可靠

- LLM是概率模型，总会有出错的时候，尤其是在长对话的时候，所以我们很有必要去有一套工程方法去约束他，这一套方法就是harness，总体有两种做法
  - 事前约束：比如你直接提前告诉他不能够修改某些文件，相当于是提前写入提示词的约束
  - 事后反馈，作完某件事就主动检查，这个就是HOOK可以做的事情

### 缓存

- 每一轮对话发动给服务器，服务器都需要把前面所有的聊天记录都在解读一遍，这样非常浪费token，所以就出现了缓存，服务器会把一些重要的内容缓存起来，下面的聊天一旦发现了需要缓存，就直接拿回前面的聊天记录，这个就叫做命中缓存，命中缓存的token消耗会少很多，大概是一二折
- 有一些服务器大概五分钟就会去掉缓存，掏钱的话就可以有更多的缓存时间，字段名可能会有一些不一样

- 有自动缓存还有主动缓存，主动缓存只需要在服务器发送的字段上加上cache_control字段就可以了

### 对话越长效果越差

- 因为对话越长，斯坦福的研究表明，LLM对开头以及结尾的记忆最为深刻，对话过长的话，不仅价格越贵，而且有很多杂乱的记忆影响思考
- 此外，如果进行压缩，也会损失很多的细节，所以的话，尽可能提前进行压缩，不要等到最后了才进行压缩

### 你的聊天记录存储在哪里

- jsonl的每一行单独拿起来都是json，聊天记录天然适合这种模式，直接往后面添加就可以了

### HOOK给claude加上规矩

  - Map_tool调用某一个已经连接的MCP sever上
  - prompt，把结构单独给一个LLM去评判，就是说引入第三者来进行
  - agent，有一个子agent来执行任务，执行完任务之后再把结果返回到主agent
- mathcer就是来匹配通配符的字符串
- 代码层面，我们可以直接告诉智能体我们要创建什么样的hook就可以了，他会自己调用工具

### sub agent：上下文管理的艺术

- 有时候我们在写一些小作文，中间需要查询资料，如果我们都是自己去查询的话，会把大脑塞满垃圾的废话而且非常浪费大脑的空间，那怎么办呢，我们就请一个外包，来帮助我们干这些事情
- 当遇到独立性比较强，或者是只看结果不注重过程的任务，就调用sub agent，他会返回最后的结果，并且，你通过开多个sub agent执行三个不同的任务，最后来进行拼接，比你一个任务来一直执行是速度快很多的
- 对于claude以及codex，他们都会内置一些sub agent，会自己判断什么时候改调用什么

### fork agent：共享上下文的艺术

- 有时候我们的一些任务需要加上前下文才能够执行，那怎么办呢，sub agent没有完整的上下文，这就有了fork agent来执行任务，他会复制完整的上下文，再加上一条新的消息，让他去执行新的任务，而且，他的最后执行结果不会返回到主agent的上下文上

- 为了触发缓存，上下文所有的这个记录都是必须是一样的，那怎么处理fork agent不会修改你的主agent的任务呢，那就只能在工具的权限做手脚了，没有授权的你是不能够操控这个工具的

# agent开发基础

### LLM的流式响应模式

-   LLM模型是概率预测模型，所以有时候我们会看到字是一个一个往外蹦的，所用的协议就是SSE，只能是单向传输，但是这样做有个缺陷，就是代码复杂度会增加
- 但是还有一个需要，你必须在客户端还有服务端都开启SSE，要不然他会把中间的消息全部都积攒完了才会传输过来，这样跟这个同步模式没有区别
- 首先先发出来头部，接着就是主要的内容，最后是结束的原因，以及消耗的token数
- gzip压缩技术是针对发送到前端文件的压缩技术，他通常需要攒够一定大小的数据才能够发送，如果你直接不关闭的话，就没有流式协议的效果，其目的就是为了让网页加速的更快，节省服务器带宽
  - 服务器带宽就是单位时间能够传输的数据总量，类似于马路，如果同一时间有很多文件传输，那就是拥挤，所以通过gzip压缩技术来节省带宽
- CDN是分布在全球的加速服务器网络，他会把网站的静态资源发送到离你最近的节点，用户请求的时候，会直接由最近的CDN返回，而不是每次返回到总部去寻找，比如我们每次访问百度的时候都是最近加速服务器，而不是跑到百度总部的服务器去询问
- 正向代理保护或隐藏了客户端，比如一些人使用科学上网去访问国外工具，但是并不知道具体访问的是谁，但是只是知道了有人访问了服务器，
- 对应的反向代理就是保护隐藏了服务端，代替后端的一堆服务器去访问外部的资源，Nginx就是一种web服务器软件，在这里充当反向代理的角色，因为用户浏览器不能直接访问后端的代码，所以需要加上这个中间的代理

### Workflow，Agent以及skill之间的区别

- workflow就是工作流，他的流程都是确定的，需要干什么事情都是确定好的，中间会调用LLM，但是LLM只是负责完成一小部分的内容
- Agent的每一步是不确定的，是有智能体自己决定的
- skill本身是不动的，它本身是由这个AI自己掉用的，不包含复杂的工作流程

### 工业界别的system prompt都有什么

- 工业级别的系统提示此并不是一个完整的字符串发送给服务端，而是拆解成了十几个碎片，有以下几个原因
  - 第一就是节省token，claude允许每一个system就是一个数组，把提示题拆解成会变的部分以及不会变的部分，就可以更高的命中缓存，那么就可以节省消费
  - 第二就是按需组合，因为有linux系统，window系统，以及mac系统，以及很多的工具，并不是所有的东西我们都需要，所以说我们只需要加上一部分就可以，这样就可以节省token，这就是前文所说的，一切都是token，都是为了上下文以及token

### 如何实现上下文压缩

- 一种是手动压缩，另外一种是自动压缩，手动压缩就是/compact，我之前自己用过一次，压缩完就会返回它压缩完的内容，自动压缩就是agent会自己判断什么时候该压缩
- 那么你怎么进行压缩呢，直接丢失前面的信息会遗漏很多的信息，所以现在基本上都是直接压缩摘要，然后把摘要方成一个新的提示词新开一个对话，继续进行聊天
- 如果有摘要不了的东西，那就直接保留，最近读过的文件以及本地回话用过的skill也会保留到这里面
- 实现上下文压缩的时候是禁止使用工具去修改你的代码的，可能会混入新的主观判断，因此要再提示词前面加上一段强约束，结尾也会在加上一道约束，因为LL M对开头以及结尾记忆是最清晰的，是真的可能会被遗忘

### claude的自动记忆系统

- 前面已经讲过，自动记忆分为用户记忆以及项目记忆，那么它是怎么判定你是否需要记忆呢，它是怎么记忆的呢，其实他就是在项目目录下的一个markdown的文本而已，记录的方向就像书本的目录加上这个正文而已，有一个文件是单独来记录这个所有的对话的索引，至于什么时候记录，全部都被写入了系统提示词，会告诉模型什么时候应该记录，什时候不应该记录。
- 去读取记忆的时候他会扫描一遍整个memory目录，然后单独起一个线程请求大模型，把清单和问题单独丢过去，挑中的记忆会被读出来，作为附加消息来插到真实的消息面前，而且会加上一组标签，来说明他是记忆内容
- claude会把记录分为四类
  - user用户的身份爱好以及一些特别的要求
  - feedback用户对工作方式的纠正
  - project当前的项目在做什么
  - reference外部系统的引用，比如登录GitHub的凭证
  - 对于feefback以及project类型还规范了正文的结构，【规则/事实】+ WHy + How to apply
- 那模型是怎么写的呢
  - claude是开了两个通道，一个来记另外一个来检查有没有遗漏的，用相对时间来记录记忆记录的时间，如果自动记忆的多了，他会把一些相同的，语意想近的东西进行合并，这样就保证了消耗的token
  - 现在已经有一些工具开始开发了，让他们可以进行记忆共享，Claude可以和codex共享记忆，这样就不需要各记各的，就类似于MC P的这个客户端与服务端类似
- codex的过程与claude的完全相反，codex是事后反馈，等关闭会话之后会先用一个小模型来记录一下值得记录的，在用一个 比较强大的模型来进行总结
- codex的记忆目录是全局的的，但是claude的记忆目录是每一个项目一个

### mem0记忆系统的实现思路

### zep记忆系统

-   使用 Zep（Graphiti）主要就两个动作：存（Add） 和 查（Search）

-   你只需要把用户说的话和对话发生的时间丢给它，它会自动在后台把这句话变成结构化的图谱记忆，并自动计算好时间先后顺序。

- 当我们需要询问的时候，直接用自然语言去查询，然后他会返回最符合当前时间状态的正确事情

### claude的权限设置

- | 模式                | 效果                                                         |
  | ------------------- | ------------------------------------------------------------ |
  | `default`           | 默认，按规则走，需要审批的工具弹窗等你点 Yes/No，最繁琐      |
  | `acceptEdits`       | 自动接受 Edit/Write 等编辑工具，省去对编辑操作的反复审批，但 Bash 命令仍需审批 |
  | `plan`              | 禁掉所有写工具，先把方案写到 plan 文件让你审，过审才动手     |
  | `bypassPermissions` | 默认允许几乎所有的操作，最省事，但是授予 AI 太多权限比较危险 |
  | `auto`              | 每次工具调用前调用另一个模型判断这次调用是否安全，安全的话自动放行。比较省事，同时兼顾安全性 |
