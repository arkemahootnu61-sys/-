# labuladong算法网站学习笔记

## 【1】LLM以及agent的核心术语以及原理

- LLM与agent之间的关系就好比与汽车的引擎以及汽车之间的管理，agent就相当于LLM的工程实现
- LLM本质上是一个概率预测，我们通常AI回答的时候看到字一个一个往外蹦，其本质上就是他在一个一个预测概率，因为参数高达几百上千亿，所以这个预测的概率就高的惊人
  - 这里就是有一个温度的概念，当温度为0是就会按照概率最大的进行生成，比较适用于编程等严谨的工作，但是如果是比较有创意的工作，温度就会调的比较高
- token就是输入以及输出的文本，所有的聊天记录就构成了上下文，但是上下文的窗口是有限的，怎么解决这个问题，一个就是压缩，另外一个就是渐进式输出

### base模型以及instruct模型

- base模型单纯就是一个文本补全的概率模型，你问他一个问题，他只会一直补全，但是instruct通过大量的指令以及回答来对数据进行微调
- RLHF是基于人类的反馈强化学习，就是让AI自己回答问题，然后人类对他的回答进行打分，然后让AI自己进行反思总结，调整自己想法

### 思维链

- 思维链：我们人类在写题的时候写出来过程会提高正确率，AI也一样，像deepseek就是第一家把思维链写出来的工具，有了思维链，后学的推导就会有更高的正确率
  - 我们有时候写题的时候会进行反思，AI有时候也会进行自我反思，本质上就是用更多的时间来换取更高精确度，这就是不同模式之间的区别，思考的快慢，之前都是单一模式以及强度参数的思考，后来才加入这种模式，可以更加省心。

### 系统提示词

- system prompt来设置人设，经常有时候我们聊一些不好的东西，他不会跟你聊天，这是因为工程师会给他提前做了一些设定，设置安全规则，
  - 用户的优先级是小于工程师的，所以我们有时候跟他说，上面的都不管用的时候，根本没有用，但是网上有这种专门越狱的.

### tools

- LLM不会上网，其本质上还有一个大脑，有很多工具他是不会调用的，所以工程师们给他写了很多工具的API接口来到system prompt，当LLM发现自己有这个工具，就会返回tool的API，然后你的系统会去调用工具，最后返回的喂给LLM，最后在回答你的问题，这就是为什么，你只是说一个hi就会消耗很多的tokens，但是后面就没有那么多了，根本原因就是第一次还发送了人设，安全规则，以及很多的tools的API，所以会耗费很多的tokens，**模型输出结构化的工具，外围程序解析结构化的工具，执行后喂给上下文**
  - 对于不同的AI，以及不同的工具，可能每一个工具的接口都不一样，就像当年的充电线一样，那怎么办呢，于是anthropic推出了自己的MCP（model context protocol,模型上下文协议）来统一规范接口的形式
  - 用MCP查天气的工具，用fastmcp就可以搞定

- **所以，agent本质上看来就把LLM + tool use加上循环**，模型在循环中不断思考,行动，观察结果，继续思考，这个模式学术山叫做ReAct（Reasoning + Acting），就是先不知道怎么做，先看看情况，再看下一步怎么做，

### skill

- 对于一些重复使用的流程，比如说我让agent识别图片整理成文档，就可以整理成文档，并且说明使用情景，下载遇到类似的情况就可以直接无脑调用了
  - 渐进式披露，通俗来讲，就是现在AI加载一个目录，说明了我有哪些skill而不全部加载，等到要使用的时候在加载进来，这样既能节省tokens

### RAG检索生成增强

- 加入你给AI一个学生手册，然后向他询问一些问题，但是你问的问题不一定跟手册的一摸一样，而且手册的关键词也有很多，那么AI是精确的理解你需要找的是什么呢，这就需要用到embedding模型了，他会将你的每一句话转化成一组语义向量（一组浮点数，比如千问就有1024维度），语义相近的文本向量距离也很近

### 自动记忆：跨会话的信息延续

- 有时候我们新开会话，但是有一些内容是需要记忆的，分为两个维度
  - 用户记忆，比如你的生日，你的风格喜好
  - 项目记忆，一些项目的坑，规范的风格，以及接口等等等
  - **总之，就是把跨对话有价值的文本沉淀下来，新的对话自动加载到上下文**

### harness(马具)：让agent更加可靠

- LLM是概率模型，总会有出错的时候，尤其是在长对话的时候，所以我们很有必要去有一套工程方法去约束他，这一套方法就是harness，总体有两种做法
  - 事前约束：比如你直接提前告诉他不能够修改某些文件，相当于是提前写入提示词的约束
  - 事后反馈，作完某件事就主动检查，这个就是HOOK可以做的事情
